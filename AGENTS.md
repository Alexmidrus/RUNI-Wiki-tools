# RUNI_Wiki: локальные инструкции для Codex

## Описание проекта
- Проект предназначен для офлайн-работы с контентом MediaWiki.
- Рабочий цикл: импорт данных с вики, локальные правки, экспорт обратно.
- Правки должны сохранять совместимость с существующей вики-разметкой и стандартами MediaWiki.

## Ключевая архитектура
- Единая точка входа: `main.py` в корне проекта.
- Все создаваемые программой файлы и папки должны находиться только в `data/`.
- Скрипты не должны записывать данные вне `data/`.

## Иерархия проекта
- Актуальная структура в корне:
  - `main.py` — единый CLI entrypoint.
  - `data/` — корень всех импортированных и служебных данных:
    - `data/templates/` — шаблоны.
    - `data/article/` — статьи и изображения.
    - `data/category/` — страницы категорий.
    - `data/source_url/` — YAML со списками URL.
    - `data/global/MediaWiki Common.js` — глобальный JS MediaWiki.
    - `data/global/MediaWiki_Common.css` — глобальные стили MediaWiki.
  - `roles/` — роли для агента:
    - `roles/role_1_python_engineer.md`
    - `roles/role_2_mediawiki_designer.md`
    - `roles/role_3_translator_eve.md`
    - `roles/role_4_copywriter_eve.md`
  - `script/project_paths.py` — единые правила путей (`project_root`, `data`, валидация output-путей).
  - `script/wiki_api.py` — общие функции MediaWiki API.
  - `script/console_ui.py` — консольный UI.
  - `script/import_template_bundle.py` — импорт одного шаблона или всех шаблонов.
  - `script/import_article_bundle.py` — импорт статьи.
  - `script/import_category_bundle.py` — импорт категории.
  - `script/import_page_urls.py` — импорт URL в YAML.
  - `script/push_templates_to_wiki.py` — выгрузка локальных шаблонов и global-стилей на MediaWiki через API.
  - `README.md` — пользовательская документация.
  - `AGENTS.md` — этот файл.

## Роли агента (`roles/`)
- Перед началом значимой задачи проверять, подходит ли одна из ролей из `roles/`.
- При профильной задаче использовать соответствующую роль как рабочую инструкцию.
- Если задача смешанная, допускается комбинирование ролей, но с явным приоритетом основной роли.
- Если роль и текущие правила конфликтуют:
  - приоритет у ограничений этого `AGENTS.md`;
  - затем у инструкции задачи пользователя;
  - затем у текста роли.

## Связи и зависимости
- `main.py` маршрутизирует команды в import-скрипты из `script/`.
- `script/project_paths.py` обязателен для всех операций, создающих файлы.
- `wiki_api.py` используется import-скриптами для HTTP/API.
- `console_ui.py` используется скриптами с интерактивным выводом статуса.

## Правила путей и файлов
- Любой `--output-root`/`--output-dir` трактуется как путь внутри `data/`.
- Абсолютный output-путь допустим только если он физически находится внутри `data/`.
- Попытки писать вне `data/` считаются ошибкой.
- При изменениях не ломать текущий формат создаваемых файлов (`*_article`, `*_category`, `*_doc`, `*_styles.css`, YAML-структура URL).

## Как ставить задачу по файлам
- При постановке задачи указывать:
  - `Путь: <file>`
  - `Назначение: что делает файл`
  - `Зачем меняем: цель`
  - `Ограничения: что нельзя ломать`
- Если файл новый, сначала добавить его в раздел `Иерархия проекта`.

## Коммуникация
- Отвечать пользователю на русском языке, кратко и по делу.
- Перед значимыми правками коротко сообщать, что именно будет изменено.
- После правок давать итог: какие файлы изменены и что проверено.

## Рабочий процесс
- Сначала собирать контекст (`ls`, `rg --files`, `rg`), потом редактировать.
- Для поиска использовать `rg`.
- Предпочитать точечные патчи и не менять несвязанные участки.
- При изменениях CLI синхронно обновлять код, `README.md` и этот файл.

## Проверка качества
- При изменениях Python-скриптов запускать минимум:
  - ``python -m py_compile main.py``
  - ``Get-ChildItem script -File -Filter *.py | ForEach-Object { python -m py_compile $_.FullName }``
- Проверять `python main.py --help` и `python main.py <command> --help`.
- Если проверки не запускались, явно указывать это в ответе.

## Безопасность изменений
- Не использовать деструктивные git-команды без явного запроса.
- Не удалять и не перезаписывать пользовательские изменения вне задачи.
- При неожиданных изменениях в рабочем дереве остановиться и уточнить действия.
